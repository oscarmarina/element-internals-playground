Análisis de BlkInput — Experto Frontend / A11y
1. Arquitectura general
La separación en tres capas es sólida:


BlkMixinInternalsBase  →  BlkMixinFormAssociated  →  BlkInput
dedupeMixin de @open-wc es la elección correcta para evitar duplicación de mixins en cadenas de herencia profundas. attachInternals() en el constructor con el guard ??= es correcto. delegatesFocus: true en el shadow root está bien aplicado y el test de :focus lo verifica.

2. Accesibilidad (A11y)
2.1 aria-errormessage — soporte inconsistente
BlkInput.ts:413, L447


aria-errormessage="${this.errorMessageText ? 'error-message-text' : nothing}"
aria-errormessage es ARIA 1.2 y el soporte real en combinaciones SR+browser sigue siendo desigual (NVDA+Firefox especialmente). El patrón más robusto es incluir el ID del error dentro de aria-describedby en lugar de confiar en aria-errormessage solo. Lo que tiene actualmente:

aria-describedby → solo apunta al mensaje de info
aria-errormessage → apunta al mensaje de error
Lo seguro sería que aria-describedby apunte a ambos cuando existan.

2.2 role="alert" + hidden — live region frágil
BlkInput.ts:371-381


<div class="error-message-text" role="alert" id="error-message-text" ?hidden="${!this.invalid}">
Dos problemas combinados:

El elemento se renderiza condicionalmente (solo cuando hay errorMessageText). Una live region debe existir en el DOM cuando la página carga para que el SR la monitorice. Si se inserta dinámicamente, muchos SRs no la anuncian.
El toggle con hidden puede no disparar el anuncio al hacerse visible — algunos SRs solo anuncian live regions cuando su contenido cambia, no cuando cambia su visibilidad.
La solución robusta: contenedor siempre en DOM, vaciar/rellenar el texto según el estado, y controlar la visibilidad con CSS en lugar de hidden.

2.3 invalid como propiedad independiente del ValidityState
BlkInput.ts:281-286


@property({ type: Boolean, reflect: true, useDefault: true })
invalid = false;
El atributo invalid es independiente del validity.valid real. Esto significa que un consumidor puede tener invalid=true con validity.valid=true o viceversa, creando una desincronía entre el estado semántico (:invalid via ElementInternals) y el visual (:host([invalid])). El CSS usa [invalid] y no :invalid, por lo que la representación visual depende de esta propiedad externa, no del estado real del constraint validation.

2.4 aria-errormessage presente cuando invalid es false
BlkInput.ts:412-413


aria-invalid="${this.invalid ? 'true' : nothing}"
aria-errormessage="${this.errorMessageText ? 'error-message-text' : nothing}"
Si errorMessageText está seteado pero invalid es false, se renderiza aria-errormessage sin aria-invalid="true". La especificación ARIA requiere que aria-errormessage solo sea válido cuando aria-invalid no es false. Algunos SRs lo ignorarán; otros lo anunciarán igual, creando comportamiento impredecible.

2.5 Placeholder como espacio — audible en algunos SRs
BlkInput.ts:420, L464


.placeholder="${this.placeholder === '' ? ' ' : this.placeholder}"
El truco del espacio con :placeholder-shown es efectivo para el floating label, pero un espacio como placeholder es anunciado como "espacio" por algunos SRs (VoiceOver en iOS especialmente). La alternativa más sólida sería usar un pseudo-elemento o el estado de :focus-within + valor vacío para controlar la posición del label sin depender del placeholder.

2.6 Alto contraste / Forced Colors — sin soporte
blk-input.scss

No hay @media (forced-colors: active). En Windows High Contrast Mode, los colores hexadecimales en custom properties no se adaptan automáticamente. Los bordes, colores de texto y fondos pueden volverse invisibles. El mínimo recomendado:


@media (forced-colors: active) {
  .field-flex {
    border: 2px solid ButtonText;
  }
  .error-message-text {
    forced-color-adjust: none;
    color: Mark;
  }
}
2.7 Touch target size
blk-input.scss:25


min-block-size: 1rem;
1rem = 16px en tamaño base. WCAG 2.5.8 (AA, WCAG 2.2) exige 24×24px mínimos. 1rem no lo cumple. Se recomienda un mínimo de 2.75rem (44px) para touch targets.

3. CSS — bugs
3.1 Opacity duplicada en estado disabled
blk-input.scss:49-60


:host(:disabled) {
  .field-block {
    opacity: 0.38;  // ← padre
  }
  .field-flex {
    opacity: 0.38;  // ← hijo dentro de .field-block
  }
}
.field-flex es hijo de .field-block. La opacidad se multiplica: 0.38 × 0.38 ≈ 0.14. El elemento disabled aparecerá mucho más oscuro/transparente de lo previsto. El ratio de contraste ya puede ser problemático a 0.38; a 0.14 es definitivamente inaceptable.

Fix: solo aplicar opacity en .field-block o en .field-flex, no en ambos.

3.2 transition shorthand incorrecto
blk-input.scss:107


transition: border-color, var(--_content-animation);
Esto genera dos transiciones separadas por coma:

border-color → sin duración (instantáneo)
var(--_content-animation) → se expande a 125ms cubic-bezier(...) sin propiedad → equivale a all
El resultado no intencionado: border-color es instantáneo (no transiciona) y todas las demás propiedades transicionan a 125ms. Lo correcto:


transition: border-color var(--_content-animation);
4. Lógica — comportamientos sutiles
4.1 _checkValidityAndSetValue no awaited en willUpdate
BlkInput.ts:341-342


override willUpdate(props: PropertyValues<this>) {
  if (props.has('value') || props.has('errorMessageText')) {
    this._checkValidityAndSetValue(this.value); // Promise sin await
  }
_checkValidityAndSetValue es async y hace await this.updateComplete internamente. Llamarla sin await en willUpdate crea una promise flotante — cualquier error será silenciado. Además, llamar updateComplete durante un ciclo de update activo puede crear scheduling inesperado en Lit.

4.2 Doble validación en connectedCallback
BlkInput.ts:329-336

connectedCallback llama _checkValidityAndSetValue después de updateComplete. willUpdate también la llama si value o errorMessageText cambian. En el primer render ambas paths se ejecutan, resultando en doble setFormValue y doble setValidity. No es catastrófico pero es ineficiente y puede disparar efectos secundarios innecesarios.

4.3 Paste / IME no marca como touched
BlkInput.ts:531-534


private _onInput({target}: Event) {
  const newValue = (target as HTMLInputElement).value;
  this.value = newValue;
  // ← no llama _markAsInteracted()
}
Un usuario que pega texto (Ctrl+V) solo dispara input, no keydown (el keydown es de Ctrl+V, que no pasa el filtro). El estado touched no se actualiza hasta que ocurra change (blur con cambio). Esto crea una ventana donde el usuario claramente ha interactuado pero touched sigue siendo false.

4.4 textarea no tiene autocomplete
BlkInput.ts:406-436

El template del <textarea> no incluye autocomplete="${this.autocomplete ?? nothing}" a diferencia del <input>. autocomplete es válido en <textarea> (WHATWG spec). La inconsistencia puede sorprender a consumidores que lo configuren.

5. Lo que funciona bien
static formAssociated = true correctamente delegado al mixin — el CE participa en form submission, reset y disabled-fieldset sin código extra en BlkInput.
formDisabledCallback con inert es una solución creativa para el problema de disabled dentro de fieldset, aunque inert tiene implicaciones más amplias que disabled.
formResetCallback limpio: restaura valor, invalid, y touched.
@media (prefers-reduced-motion: reduce) en las animaciones del label — correcto.
Tests de a11y con chai-a11y-axe + los casos de label externo, fieldset disabled — cobertura sólida para los escenarios principales.
CSS lógico (inline-size, block-size, inset-block-start, inset-inline-start) — preparado para escritura RTL/vertical.
El patrón de label externo: cuando no hay this.label interno, el shadow input no tiene id, y aria-label se propaga desde this.labelText (que lee this.labels vía ElementInternals). Esto resuelve elegantemente el problema del shadow DOM encapsulation para labels externos.
